Acredito que uma boa abordagem, não tão complexa e que resolve a maioria dos problemas, é a seguinte:

Temos basicamente 3 ramificações principais e mais algumas auxiliares. As principais seriam:

  • main - Branch principal do sistema
  • release - Todos os PRs são "mergeados" na release, onde são agrupados para criar novas versões do sistema e, então, são enviados para a main.
  • staging - Ambiente de teste, onde novas features são validadas antes de irem para a release

Vou explicar melhor o fluxo de trabalho:

Sempre que temos uma nova task, devemos criar uma branch auxiliar baseada na staging, e não na main. Essa branch pode iniciar com um nome padrão, por exemplo "feature/resto-do-nome", "fix/resto-do-nome", "chore/resto-do-nome"...

Ao finalizar o desenvolvimento da task, devemos criar um PR para ser feito code review pela equipe.

Se for aprovado pela equipe, o PR deve ser "mergeado" na staging, onde poderá ser testado.

Em caso de hotfix, quando há um bug crítico no sistema, acredito que devemos criar uma branch a partir da main. Após ser aprovada pelo code review, pode ser "mergeada" diretamente na main. Sei que isso é meio arriscado, não passar pelo staging, mas normalmente hotfixes são erros críticos que devem ser resolvidos com urgência.
